# Plan: Custom Provider Management Implementation

## Reference: Legacy modelRouter Project

The existing `/home/petrouil/Projects/modelRouter/` (Python/FastAPI) provides a proven design pattern to adopt:

- **Provider registry** with CRUD + model discovery
- **Model ID pattern** - not used directly; Clawdock stores `model_key` separately
- **Auth handling** - provider-level API keys stored (encrypted) and sent per-request
- **Discovery implementations** per provider type (Ollama, OpenRouter, OpenAI, Anthropic)
- **Sync algorithm** - iterate enabled providers, call their `list_models()`, upsert

Key files: `app/models.py` (Provider, ModelEntry), `app/registry.py` (sync logic), `app/providers/` (per-vendor clients).

---

## Current State (Clawdock)

**Existing schema:**
- `providers`: `id`, `display_name`, `base_url`, `auth_type` (currently: `api_key`/`bearer`/`none`), `enabled`, `supports_model_discovery`, timestamps
- `provider_models`: `id (uuid PK)`, `provider_id`, `model_key`, `display_name`, `enabled`, `sort_order`, `metadata_json`
- Seeded providers: openai, anthropic, google, ollama, openrouter, custom
- Seeded models: per provider preset list

**Existing API (read-only):**
- `GET /api/providers` → list all
- `GET /api/providers/:id/models` → models for provider

**Existing frontend:**
- `providers.js` renders static provider cards with model counts

**Gaps:**
- No provider CREATE/UPDATE/DELETE
- No model discovery
- No model enable/disable API
- No provider-level API key storage (agents have only `telegram_api_key_encrypted`)
- No built-in protection flag

---

## Desired Capabilities

1. Users can **create custom providers** with display name, base URL, auth protocol, optional API key, and enable discovery
2. Users can **edit** and **delete** custom providers; built-ins are protected from deletion but editable
3. **Model discovery**: fetch model list from provider API and store in `provider_models`
4. **Enable/disable** individual models per provider from UI
5. Provider API keys stored encrypted; used for discovery and as default for new agents
6. Manual refresh via "Refresh Models" button; no auto-scheduled discovery
7. Integration with agent creation: validate selected model exists and is enabled; optionally override provider API key

## Design Decisions

### 1. Provider Authentication & Keys

**Choice:** Store `api_key_encrypted TEXT` column in `providers`. Use AES-256-GCM with server's `secret_key` from config. Provide helper functions `Encrypt(plaintext)`, `Decrypt(ciphertext)`.

The API never returns encrypted key except masked display using `MaskSecret()`.

**Default behavior:** If agent creation uses a provider with an API key, the agent's `telegram_api_key_encrypted` can default to the provider's key (or agent inherits it at creation). For flexibility, allow agent-level override.

### 2. Built-in Provider Protection

Add boolean column `is_builtin BOOLEAN DEFAULT 0 NOT NULL`. Set to 1 on seeded provider rows via migration. Prevent DELETE when `is_builtin=1`. Allow PATCH edits to all other fields.

### 3. Auth Protocol Values

Reuse `auth_type` column (already exists). Values: `none`, `api_key`, `bearer`. Keep simple. (`basic`, `custom` deferred.)

Interpretation:
- `none` - no Authorization header (Ollama)
- `api_key` - `Authorization: Bearer {key}`
- `bearer` - same as `api_key` (explicit naming)

Provider discovery uses the stored `api_key_encrypted` when `auth_type` is `api_key` or `bearer`.

### 4. Model Identification

Keep current schema: `provider_models.id` is UUID primary key. The `model_key` is the identifier agents reference (e.g., `"gpt-4o"`, `"llama3"`). Add unique constraint `UNIQUE(provider_id, model_key)` to prevent duplicates.

The agent's `model_id` field stores the `model_key` string (not the UUID). This matches current code: `agents.model_id` used directly.

### 5. Discovery ID Format

Do not adopt composite string IDs from modelRouter. Use our `id` as UUID; `model_key` as human-friendly key. Expose both in API.

### 6. Discovery Implementation

Each provider type gets a `DiscoverModels(baseURL, apiKey) ([]string, error)` function. Discovery maps:
- Ollama → `GET /api/tags` → `models[].name`
- OpenAI → `GET /v1/models` → `data[].id`
- Anthropic → `GET /v1/models` → `data[].id`
- Google → `GET /v1/models` → `models[].name` (or `data[].name` depending; check docs)
- OpenRouter → `GET /api/v1/models` → `data[].id`
- Custom → no discovery; must add models manually

Return model keys (strings). The handler stores them: for each key, check if exists (`SELECT id FROM provider_models WHERE provider_id=? AND model_key=?`). If exists → update `display_name` and ensure `enabled=true`. If new → insert new row with UUID and key.

Deletion of stale models: optional. Safer to only add/enable; manual removal if needed.

### 7. API Encryption

Reuse existing `security.HashSecret`? That's one-way; we need reversible. Implement AES-GCM:
```go
func Encrypt(plaintext string, secret []byte) (string, error)
func Decrypt(ciphertext string, secret []byte) (string, error)
```
Store base64-encoded ciphertext. Load secret from config at startup.

### 8. Database Migrations

Since clawdock uses code-based migrations (schema in `migrate()`), we'll ALTER TABLE and then `UPDATE` seeded providers:

```go
// In migrate() or separate function called after Init
ALTER TABLE providers ADD COLUMN api_key_encrypted TEXT;
ALTER TABLE providers ADD COLUMN is_builtin BOOLEAN DEFAULT 0 NOT NULL;
-- auth_type already exists, keep it
CREATE UNIQUE INDEX IF NOT EXISTS idx_provider_model_key ON provider_models(provider_id, model_key);

// Then update seeded rows
UPDATE providers SET is_builtin=1, auth_type='api_key' WHERE id='openai';
UPDATE providers SET is_builtin=1, auth_type='api_key' WHERE id='anthropic';
UPDATE providers SET is_builtin=1, auth_type='api_key' WHERE id='google';
UPDATE providers SET is_builtin=1, auth_type='none' WHERE id='ollama';
UPDATE providers SET is_builtin=1, auth_type='bearer' WHERE id='openrouter';
UPDATE providers SET is_builtin=1, auth_type='api_key' WHERE id='custom';
```

Note: Google may use API key query param; we'll handle in discovery. Custom provider has no discovery endpoint.

## Implementation Phases

### Phase 1: Database & Encryption

**Files:** `internal/database/database.go`, `internal/security/crypto.go` (new)

- Add columns to `providers` in `migrate()` using ALTER TABLE IF NOT EXISTS
- Add unique index on `provider_models(provider_id, model_key)`
- Upgrade seeded data: set `is_builtin=1` per provider ID with correct `auth_type`
- Create `crypto.go` with `EncryptAESGCM`, `DecryptAESGCM` using 256-bit key from config secret
- Update config loader to expose secret key bytes

### Phase 2: Provider CRUD Handlers

**File:** `internal/handlers/provider_handlers.go`

Methods on main Handler:
```go
func (h *Handler) CreateProvider(w http.ResponseWriter, r *http.Request)
func (h *Handler) GetProvider(w http.ResponseWriter, r *http.Request)
func (h *Handler) UpdateProvider(w http.ResponseWriter, r *http.Request)
func (h *Handler) DeleteProvider(w http.ResponseWriter, r *http.Request)
func (h *Handler) RefreshProviderModels(w http.ResponseWriter, r *http.Request)
```

Auth: All require auth if `cfg.Security.RequireAuth` is true; currently disabled by default.

Create:
- Decode JSON: `{display_name, base_url, auth_type, api_key?, enabled?, supports_model_discovery?}`
- Validate `auth_type` is allowed set
- Validate URL scheme (http/https)
- If `api_key` provided → encrypt → store; else store NULL
- Insert with UUID, `is_builtin=false`
- Audit log `create_provider`

Get:
- SELECT provider row (no api_key in response, or masked if needed)
- Return JSON with fields minus sensitive data

Update:
- PATCH partial update; allowed: display_name, base_url, auth_type, api_key_encrypted (replace), enabled, supports_model_discovery
- If updating `api_key`, replace encrypted value
- Reject if `is_builtin && field not allowed`? Allow all except maybe `id` is_builtin remains.
- Audit log `update_provider`

Delete:
- Check `is_builtin`, if true → 403 Forbidden
- DELETE FROM providers WHERE id = ?
- Audit log `delete_provider`

RefreshModels:
- Call `providers.DiscoverAndUpsertModels(h.db, providerID)`
- On success: return `{models_added: N, models_updated: M}`
- Audit log `refresh_provider_models`

### Phase 3: Model Toggle Endpoint

**File:** `internal/handlers/provider_handlers.go`

`UpdateProviderModel(w, r)`:
- Path param `id` = provider_model UUID
- Body: `{enabled: boolean}`
- UPDATE provider_models SET enabled = ? WHERE id = ?
- Audit log `update_provider_model`

Alternative: `PATCH /api/providers/:providerID/models/:modelID` but using model UUID simpler and decoupled.

### Phase 4: Frontend Provider Management

**HTML (`index.html`):**
- In Providers view (after "X providers" header), add "Create Provider" button
- Add modal `#create-provider-modal` with form: name, base URL, auth type (select), API key (password, optional), discovery checkbox
- Edit modal reused; pre-fill; API key field shows placeholder "Leave blank to keep existing"
- Add small badge on provider card for built-in: "Built-in" (non-clickable delete)

**JS (`providers.js`):**
- Add `createProvider(formData)`, `updateProvider(id, formData)`, `deleteProvider(id)`, `refreshModels(id)`, `toggleModel(modelId, enabled)`
- Provider card HTML:
  - Action bar: Edit button, Refresh button, Delete (if custom)
  - Model list rows: checkbox for `enabled`, model name + key
  - On checkbox change → call toggleModel, show toast
- Provider form modal:
  - Auth type select change → show/hide API key field
  - Submit: if create → call api.providers.create; then if discovery checked, call refreshModels; close & refresh
- Loading states: show spinner on model list during refresh

**API client (`api.js`):**
- Extend `api.providers` with create, update, delete, refreshModels
- Add `api.providerModels.update(modelId, {enabled})`

### Phase 5: Discovery Engine

**Package `internal/providers/`:**

`discovery.go`:
```go
type Discoverer interface {
    Discover() ([]string, error)  // list of model keys
}
type OllamaDiscoverer struct{ baseURL string }
type OpenAICompatibleDiscoverer struct{ baseURL, apiKey string }
type OpenRouterDiscoverer struct{ baseURL, apiKey string }
```
Implement `Discover()` for each with appropriate HTTP GET and JSON parsing.

`registry.go`:
```go
func DiscoverAndUpsertModels(db *sql.DB, providerID string) error
```
- Fetch provider record (auth_type, base_url, api_key_encrypted, supports_model_discovery)
- If `!enabled || !supports_model_discovery`, return error
- Build discoverer based on `auth_type`:
  - `none` → OllamaDiscoveror (no key)
  - `api_key` / `bearer` → OpenAICompatible (or provider-specific; but openai/anthropic/google similar)
  - `custom` → return error "discovery not supported"
- If provider is 'openrouter', use OpenRouterDiscoverer
- Call discoverer.Discover()
- For each modelKey returned:
  - Sanitize: trim spaces
  - Check existing: SELECT id FROM provider_models WHERE provider_id=? AND model_key=?
  - If exists → UPDATE enabled=true, display_name=modelKey (or nicer), updated_at=NOW()
  - If not exists → INSERT with new UUID, display_name=modelKey, enabled=true, sort_order=0
- Return summary

### Phase 6: Agent Form Integration

**`agent-form.js`:**
- When provider selection changes, fetch provider details (maybe already have state)
- If provider `auth_type` in ['api_key','bearer'] and provider has stored `api_key` (not shown), we can prefill a hint. Or just accept agent-level key.
- Model dropdown populates from `providerModels` state filtered by provider_id. After provider creation discovery, models appear automatically due to state refresh.

### Phase 7: Backend Model Validation

**`agent_handlers.go`:**
- In CreateAgent: after INSERT, check provided `model_id` exists in `provider_models` for that provider and `enabled=1`. Return error if invalid.
- In UpdateAgent: if `model_id` changes, validate same.

No need to block existing agents; they already have model IDs. New agents must reference valid models.

### Phase 8: Testing

- Unit tests for discovery mocks (using `httptest.Server`)
- Handler tests for provider CRUD (ensure encryption, audit)
- Database migration test: ensure columns added without data loss

## Example: Using the Example Ollama Server

To support `https://mainframeollama.petrouil.com/`:

- Create provider with `auth_type = "none"`, `base_url = "https://mainframeollama.petrouil.com"`, `supports_model_discovery = true`
- Click "Refresh Models" → hits `GET https://mainframeollama.petrouil.com/api/tags`, parses model names, stores them
- Agent creation can select those models; no API key needed

For OpenRouter:
- `auth_type = "bearer"`, `base_url = "https://openrouter.ai"`, provide API key
- Discovery: `GET https://openrouter.ai/api/v1/models` with `Authorization: Bearer <key>`

For custom local LLM proxy with OpenAI-compatible API:
- `auth_type = "api_key"`, `base_url = "http://localhost:8000"`, key as needed
